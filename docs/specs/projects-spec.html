Projects

Design specification — replacing the hardcoded "Build" tab with a generic multi-project tracker that works for any large staged commitment.

What's changing and why

The current "Build" tab is a well-designed tracker for a fixed-price home construction contract — stage payments, variations, and outside extras. The problem is it's hardcoded to that single scenario and assumes every user is building a house.

Current — Build tab
One hardcoded project
Stage names fixed (Slab, Frame, etc.)
Assumes construction context
No way to add a second project
Irrelevant to renters / owners
Proposed — Projects tab
Multiple projects supported
Stage names fully customisable
Six project type templates
Works for any large commitment
Relevant to every household
✅ What stays the same: The core data structure (stages, variations, extras) is solid and transfers directly. The visual stage timeline and table are kept. Only the hardcoded assumptions are removed.

Data Model

The current single buildContract object becomes a projects[] array. Each project is self-contained.

ℹ️ Variations and extras keep their existing structure unchanged. They're simply moved from state.buildContract to state.projects[i] so each project has its own independent set.

Migration from Build

Users who already have Build data need their data migrated automatically — they should never have to re-enter anything.

Migration logic (run once on load)

Old fieldMaps toNotes
state.buildContract.totalprojects[0].contractValueBecomes the first project's contract value
state.buildContract.stages[]projects[0].stages[]Direct copy — same structure
state.buildContract.variations[]projects[0].variations[]Direct copy
state.extras[]projects[0].extras[]Direct copy
—projects[0].nameDefault: "My Build"
—projects[0].typeDefault: "construction"
—projects[0].statusDefault: "active"
⚠️ Keep old keys alive briefly: Don't delete state.buildContract and state.extras immediately. Keep them for one or two releases as a fallback, then clean up. This protects users who downgrade or reload an older cached version.
✅ Migration is silent: The user sees their existing build data exactly as before — just now inside a project card named "My Build". No notification or wizard needed.

What to do with Furniture & Appliances

The current data model also has state.furniture[] and state.appliances[] as separate arrays. These are logically "extras" that belong to the build project. They should be migrated into projects[0].extras[] with a category tag (category: 'furniture' / 'appliance') so they're visible in the Extras tab without losing their distinction.

ℹ️ Extras categories: Adding an optional category field to extras (e.g. Landscaping, Solar, Appliances, Furniture, Legal, Other) allows filtering within the Extras tab without requiring separate sub-tabs.
