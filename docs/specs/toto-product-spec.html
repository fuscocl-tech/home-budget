🐕 Toto

Family household management — routines, finances, and kids all in one place.

PWA + iOS (Capacitor) Vanilla JS · Firebase · localStorage Single file · ~21,500 lines Spec v1.0 · May 2026

Product Vision

Toto is a family household management app that brings together two distinct but connected experiences: a financial and household planning tool for adults, and a fun, age-appropriate activity tracker for children.

Adults manage budgets, bills, routines, chores, meals, vehicles, documents, and events. Children log in on the same device, see only what's relevant to them — their routines, chores, and calendar — and earn points toward prizes they can redeem with parent approval.

The single shared device model is a core design constraint. Toto assumes families share an iPad or phone and routes each user to the right experience based on device profile and PIN.

Target Users

👨‍👩‍👧 Parents / Adults

Manage household finances, set up routines and chores, approve completions and prize redemptions, schedule events, track documents and vehicles.

🧒 Children (ages 1–16)

Complete daily routines and chores, earn points, view their calendar, redeem prizes. Experience adapts to age bracket automatically.

📱 Shared Family Device

Single iPad or phone used by the whole family. Profile selector on each open routes each person to their own view.

Point Balance Calculation

balance = Σ(approved completions × chore.points)

Routines

Adult-created task sequences that children complete on a schedule. Supports rich recurrence, per-child assignment, and individual exceptions.

Routine Record

Routine {
  id, name, emoji
  ownerType: 'adult' | 'household'
  ownerId, sharedWith: uid[]
  steps: [{
    id, label, emoji, points, durationMin, notes
  }]
  pointsPerCompletion   // bonus on full completion
  triggerTime           // e.g. "06:00" — determines time-lock on child view
  recurrence: {
    type: 'daily' | 'weekdays' | 'weekends' | 'specific_days' | 'interval' | 'one_time'
    days?: number[]         // [0–6] for specific_days (0 = Sun)
    intervalDays?: number   // for interval type
    startDate: ISODate
    endDate?: ISODate        // absent = recurs indefinitely
  }
  skippedDates: ISODate[]
  pausePeriods: [{ from: ISODate, to: ISODate, reason?: string }]
  lastEditedAt: ISO8601
}

Recurrence Types

TypeDescriptionDefault
dailyEvery day✓ Pre-selected
weekdaysMonday – Friday only
weekendsSaturday – Sunday only
specific_daysAdult selects 1+ days from day-picker (Su Mo Tu We Th Fr Sa)
intervalEvery N days from start date (e.g. every 2 days)
one_timeSingle specified date only

Recurrence Engine

Shared pure function used by Routines, Child Events, and Planner events:

_recurrenceMatchesDate(recurrence, dateStr)
  → checks startDate / endDate bounds
  → evaluates type (daily/weekdays/weekends/specific_days/interval/one_time)
  → returns boolean

_routineMatchesDate(routine, dateStr)
  → checks pausePeriods (date in range? → false)
  → checks skippedDates (date in array? → false)
  → calls _recurrenceMatchesDate()

Skip a Single Day

  • "Skip this day" action on any routine instance from adult view
  • Date added to skippedDates[]
  • Adult calendar shows muted/strikethrough on skipped dates
  • Can be un-skipped from same menu
  • Child view: skipped routines simply don't appear — no explanation shown to child

Pause a Routine

  • "Pause routine" in edit menu — select from/to date range + optional reason label
  • Multiple non-overlapping pause periods supported
  • Resumes automatically when pause period ends
  • Can be ended early from routine detail screen
  • Child view: paused routines don't appear during the window

Routine Assignment

RoutineAssignment {
  id
  routineId
  childId
  completionState: {
    'YYYY-MM-DD': [stepId, ...]   // steps completed on that date
  }
}

Each child has an independent RoutineAssignment — one child completing a routine does not affect another's record.

Daily Reset

Completion state resets each day. Reset time: midnight (configurable per household — suggested option of e.g. 6am). Historical completion data is preserved in the completionState map; only the current day's key is used for today's display.

Recurrence UI (Adult)

  • "Repeat" control in routine create/edit flow
  • Specific days: day-of-week picker appears (Su Mo Tu We Th Fr Sa, toggleable)
  • Optional end date picker, labelled "Ends on… (optional)"
  • Plain-language summary: "Repeats every Monday, Wednesday and Friday starting 1 May"

State Structure

All app data lives in a single state object. Persisted to localStorage and Firestore on every mutation via saveData(state).

state {
  // Household
  householdProfile: {
    members: [{ role, name, age, emoji }]
    pets: [], cars: 1
    invites: [], authorizedUsers: [{ uid, name, email }]
  }

  // Budget / Finance
  budget: {
    income: [{ name, amount, frequency }]
    expenses: [{ name, category, amount, frequency }]
    actuals: { 'YYYY-MM': { expenseId: amount } }
  }
  bills: [{ id, name, amount, dueDay, category }]
  netWorth: { assets, liabilities, snapshots, target }
  goals: [{ id, name, targetAmount, currentAmount, deadline }]

  // Planning
  planner: { events: [{ id, title, category, date, time, recurrence, memberIds }] }
  meals: { plan, shopping, lunchbox }

  // Household
  vehicles: [{ id, name, plate, rego, insurance }]
  documents: [{ id, name, provider, expiryDate }]
  maintenance: [{ id, name, dueDate, provider, frequency }]

  // Kids zone
  routines: [Routine]
  routineAssignments: [RoutineAssignment]
  childEvents: [ChildEvent]
  kids: {
    profiles: [{ id, name, age, emoji, pinHash, dob }]
    chores: [{ id, name, emoji, assignedTo, points, frequency }]
    prizes: [{ id, name, emoji, pointCost }]
    completions: [{ id, kidId, choreId, status, ts }]
    redemptions: [{ id, kidId, prizeId, status, ts }]
    notifications: [{ id, kidId, message, read, ts }]
  }

  // App state
  settings: { householdName, adultName, email, notifStyle, ... }
  activityLog: [{ action, detail, ts }]
  categoryGroups: [{ id, name, icon, categories }]
}

Storage & Sync

localStorage (Primary)

  • Key: home_finance_v1
  • Full state JSON — synchronous read/write
  • Fast, offline-capable, always available

Firebase Firestore (Cloud)

  • Document path: /households/{householdOwnerUID}
  • Real-time listener via _startFirestoreSync()
  • Bi-directional: Firestore changes call renderAll(); local changes write back
  • Fallback: app works fully offline; syncs when reconnected

Data Migrations

_applyMigrations(state) runs on every load and handles:

  • Backfilling recurrence, skippedDates, pausePeriods on legacy routines
  • Migrating old planner event recurring string to rich recurrence object
  • Deduplicating kid profiles by name (keeps most recent)
  • Converting legacy adults/children counts to rich members[] model
  • Initialising missing top-level keys (childEvents, routineAssignments, etc.)

Auth & Households

  • Firebase Auth with Google Sign-In provider
  • Guest mode available — full offline exploration without sign-in
  • Household owner UID stored in localStorage('toto_household_owner')
  • Additional adults invited via token system (stored in householdProfile.invites)
  • Invited adults are added to authorizedUsers[] after accepting invite
  • Multiple authorized users share the same Firestore household document

Rendering Architecture

  • renderAll() — re-renders Today + whichever tab is currently active
  • activateTab(tab) — switches visible panel, updates nav active states, calls renderAll()
  • Each screen has a dedicated render function (renderBudget(), renderKids(), etc.) that writes HTML into its content div via innerHTML
  • Static section headers (Wallet / Plan / Home) live in the HTML and are shown/hidden by activateTab — never injected dynamically, so renderAll() cannot wipe them
  • Child view is a separate full-screen overlay (#child-view-overlay) — swapped in when a child profile is active
